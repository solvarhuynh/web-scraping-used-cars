# Data Processing Workflow

This document outlines the end-to-end pipeline for scraping, cleaning, and processing used car data from various sources. The entire process should be orchestrated by a main script (`run_pipeline.R`).

---

### Project Directory Structure

To keep the project properly organized, scripts and data files must be strictly placed into specific subdirectories based on their function:

-   **`script/` directory**: Must be divided into:
    -   `script/scrap/`: For all raw data scraping scripts (e.g., `scrap_chotot.R`).
    -   `script/clean/`: For all data cleaning and standardization scripts (e.g., `clean_chotot.R`).
    -   `script/realtime/`: For scripts handling continuous/real-time database updates.
-   **`data/` directory**: Must be divided into:
    -   `data/raw/`: For the initial raw CSV outputs generated by the scraping scripts.
    -   `data/clean/`: For the processed and standardized CSV outputs generated by the cleaning scripts.

---

### Step 1: Data Scraping

1.  **Objective**: Scrape raw data from each target website.
2.  **Scripts**: For each website, a dedicated R script named `scrap_{website_name}.R` will be created and placed in the `script/scrap/` directory.
3.  **Rules**: Each script must adhere to the guidelines defined in `scrap_rule.md`.
4.  **Output**: Upon successful execution, each script will generate a raw data file named `data_{website_name}_raw.csv` in the `data/raw/` directory.
5.  **Logging**: The script must log the start time, end time, number of records scraped, and any errors encountered.

### 1.1. Pagination/Load More Logic

-   **For websites using HTML parsing (e.g., Bán Xe Hơi Cũ, Oto.com.vn)**:
    -   Identify HTML elements responsible for pagination (e.g., "Next Page" button, page number links, or "Load More" button).
    -   If a "Load More" button exists, trigger it (using libraries like `RSelenium` if JavaScript interaction is needed) and wait for new content to load.
    -   If page number links or "Next Page" buttons exist, extract the URL of the next page and repeat the scraping process.
    -   Ensure a stop condition is implemented to prevent infinite loops (e.g., no next page link found, or encountering an already processed listing).
    -   For websites with "Load More" buttons, duplicate prevention logic should be implemented to avoid re-scraping items already processed in the current session.

---

### Step 2: Data Cleaning

1.  **Objective**: Clean and standardize the raw data from each source.
2.  **Scripts**: For each raw dataset, a corresponding cleaning script named `clean_{website_name}.R` will be executed. These scripts are located in the `script/clean/` directory.
3.  **Rules**: The cleaning process must follow the rules specified in `clean_rule.md`.
4.  **Output**: Each script will produce a clean data file named `data_{website_name}_clean.csv` in the `data/clean/` directory.
5.  **Logging**: Log the start, completion, number of rows processed, and any data quality issues found.

---

### Step 3: Database Ingestion (Individual Sources)

1.  **Objective**: Convert each cleaned CSV file into its own SQLite database for intermediate storage and initial deduplication.
2.  **Script**: A script like `init_database.R` will process each `data_{website_name}_clean.csv` file from the `data/clean/` directory.
3.  **Process**: For each CSV, it will create a new SQLite database file named `data_{website_name}.db` in the `data/init_db/` directory. The data from the CSV will be imported into a `car_listings` table within this new database, ensuring the `url` column is a `PRIMARY KEY UNIQUE`.
4.  **Output**: Multiple individual SQLite database files (`data/init_db/*.db`), each containing listings from a single source.
5.  **Benefits**: This approach is more robust and memory-efficient, especially for large datasets, as it leverages SQLite's capabilities for initial data storage and uniqueness enforcement per source.

---

### Step 4: Consolidated Master Database Creation

1.  **Objective**: Consolidate all individual source databases into a single, master SQLite database.
2.  **Script**: A script named `merge_data.R` will be used for this step.
3.  **Process**: The script will read all `data_{website_name}.db` files from the `data/init_db/` directory. It will then extract all `car_listings` from these individual databases and insert them into a single `car_listings` table within `data/master_data.db`. The `url` column in `data/master_data.db` will also be a `PRIMARY KEY UNIQUE`, ensuring final deduplication across all sources.
4.  **Output**: A single, consolidated SQLite database file named `data/master_data.db`.
5.  **Note**: This step creates the definitive master dataset, leveraging the database's integrity constraints for efficient and robust merging and deduplication.

---

### General Rule: Console Output (Progress Bar)

-   **Progress Indicator**: During the execution of lengthy tasks (e.g., iterating through multiple pages, URLs, or cleaning thousands of rows), the scripts must display a dynamic progress bar in the console.
-   **Percentage & ETA**: You should utilize R packages such as `cli` or `progress` to render a visual bar that shows the current completion percentage (0% to 100%) and the Estimated Time of Arrival (ETA).

### Step 5: Real-time Execution

1.  **Objective**: Continuously update the database with the latest car listings in near real-time.
2.  **Rules**: The process must strictly follow the rules defined in `realtime_rule.md`.
3.  **Process**: The script is scheduled to trigger at short intervals, scraping only the newest data from the first pages, and injecting it directly into the `.db` file using delta fetching logic.

---

### General Rule: Logging

-   **File**: All process notifications and events must be appended to the `log.txt` file located in the project's root directory.
-   **Format**: Each log entry should be timestamped and include the script name and a descriptive message (e.g., `[YYYY-MM-DD HH:MM:SS] [scrap_chotot.R] - INFO: Successfully scraped 520 records.`).

# Code Flattening and Logic Simplification Rules

### 1. Use Repo-Relative Paths

Use paths relative to the repository root so the project can run on Windows, macOS, and Linux without editing scripts.

-   **Standard**: Use repo-relative paths such as `web_scraping/data/raw/...`.
-   **Standard Project Root**: the repository root that contains `app.R`, `web_scraping/`, `machine_learning/`, and `insights/`.
-   **Ví dụ**:
    -   *Preferred*: `source("web_scraping/script/utils.R")` or `file.path("web_scraping", "data", "raw", "data_banxehoicu_raw.csv")`
    -   *Avoid*: machine-specific absolute paths.

### 2. Execution Structure Flattening

Các script điều khiển (Orchestrator) không nên bao bọc toàn bộ logic trong một hàm `main()` hoặc `run_xxx()`. Code cần được triển khai để chạy tuyến tính từ trên xuống dưới.
Orchestrator scripts should not wrap the entire logic inside a `main()` or `run_xxx()` function. Code should be implemented to run linearly from top to bottom.

-   **Standard**: Remove function definitions that wrap the main execution logic.
-   **Benefit**: Easier line-by-line debugging and monitoring of variables in the Global environment without worrying about function scope.
-   **Ví dụ**:
    -   *Preferred*: Write `tryCatch` blocks, `for` loops, and data processing directly at the script level.
    -   *Avoid*: `run_pipeline <- function() { ... }; run_pipeline()`

### 3. Replace Redundant Utility Functions with Direct Commands

If an operation can be performed using 1-2 lines of standard R or Tidyverse code, prioritize direct implementation over creating helper functions.

-   **Logging**: For simple scripts, use `cat()` directly to write to `LOG_FILE` instead of a `log_msg()` function.
    -   *Template*: `cat(paste0("[", Sys.time(), "] - INFO: Merging records...\n"), file = LOG_FILE, append = TRUE)`
-   **Directory Checks**: Use `if (!dir.exists(PATH)) dir.create(PATH, recursive = TRUE)` directly.

### 4. "Ask Before Filling" Principle

During code maintenance or upgrades:
-   If encountering unclear parameters (e.g., new links, tokens, or unannounced directory structure changes), AI/Programmers **must not make assumptions**.
-   Explicit confirmation must be sought from the project manager before populating values in the code.

# Data Processing Workflow

This document outlines the end-to-end pipeline for scraping, cleaning, and processing used car data from various sources. The entire process should be orchestrated by a main script (`run_pipeline.R`).

---

### Project Directory Structure

To keep the project properly organized, scripts and data files must be strictly placed into specific subdirectories based on their function:

-   **`script/` directory**: Must be divided into:
    -   `web_scraping/script/scrap/`: For all raw data scraping scripts (e.g., `scrap_chotot.R`).
    -   `web_scraping/script/clean/`: For all data cleaning and standardization scripts (e.g., `clean_chotot.R`).
    -   `web_scraping/script/realtime/`: For scripts handling continuous/real-time database updates.
-   **`data/` directory**: Must be divided into:
    -   `web_scraping/data/raw/`: For the initial raw CSV outputs generated by the scraping scripts.
    -   `web_scraping/data/clean/`: For the processed and standardized CSV outputs generated by the cleaning scripts.

---

### Workflow Steps

**Step 1: Data Scraping**

1.  **Objective**: Scrape raw data from each target website.
2.  **Execution**: Run scripts in `script/scrap/` following `scrap_rule.md`.
3.  **Output**: Generate `data_{website_name}_raw.csv` in `data/raw/`.

**Step 2: Data Cleaning**

---
1.  **Objective**: Clean and standardize the raw data from each source.
2.  **Execution**: Run scripts in `script/clean/` following `clean_rule.md`.
3.  **Output**: Generate `data_{website_name}_clean.csv` in `data/clean/`.

**Step 3: Database Ingestion (Individual Sources)**

1.  **Objective**: Convert each cleaned CSV file into its own SQLite database for intermediate storage and initial deduplication.
2.  **Script**: A script like `init_database.R` will process each `data_{website_name}_clean.csv` file from the `web_scraping/data/clean/` directory.

**Step 4: Consolidated Master Database Creation**

1.  **Objective**: Consolidate all individual source databases into a single, master SQLite database.
2.  **Script**: `merge_data.R`.

**Step 5: Real-time Execution**

1.  **Objective**: Continuous updates via `realtime_rule.md`.

---

### General Rule: Logging

-   **File**: All process notifications and events must be appended to `web_scraping/log.txt`.
-   **Format**: Each log entry should be timestamped and include the script name and a descriptive message (e.g., `[YYYY-MM-DD HH:MM:SS] [scrap_chotot.R] - INFO: Successfully scraped 520 records.`).
